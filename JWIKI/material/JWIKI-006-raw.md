## Features principales

1. **Sistema de doble eje ortogonal (Audience × WorkMode):** 2 audiencias (User/Developer) × 3 modos (Chat 14 herramientas solo-lectura / Edit 30 herramientas totales / Plan 21 herramientas de planificación). Herramientas y system prompts cambian dinámicamente según el modo. Cambio automático Edit→Plan cuando se detecta tarea compleja, con aprobación del usuario.
2. **Pipeline Agent de 5 fases:** setup → validación de intención → construcción dinámica de contexto → loop principal (compresión, API call, SSE stream, ejecución de tools) → cierre. Cancelable en cualquier punto vía `CancellationToken`.
3. **Clasificación de intención en 3 capas:** reglas (regex, ~90% de casos, cero latencia) → contexto (últimas 4 acciones del asistente) → LLM ligero de fallback. Logging dedicado vía `debug_logger`.
4. **Snapshot engine con versionado tipo Git:** árbol de snapshots file-level, rollback atómico, branching, merge, GC de 3 fases. Multi-Agent sandbox paralelo con merge posterior (`snapshot_engine/multi_agent/{sandbox.rs, merge.rs}`).
5. **Sub-agent delegation + scheduler con grafo de dependencias:** `CreateTask`/`UpdateTask` (blocked_by), `JoinSet` streaming, paralelización automática de tareas sin dependencias, timeout 5 min. Emisión de eventos Tauri `chat-content`, `bg-task-done`, `subagent-updated` (con throttle 2s).
6. **Soporte multi-modelo (20+ LLMs):** DeepSeek, Claude, GPT, Gemini, Qwen, 豆包, MIMO, etc. Doble formato API: OpenAI y Anthropic. Configuración con múltiples perfiles (Profile) y escritura atómica (tmp+rename).
7. **Sistema de memoria y compresión:** memoria global + memoria de proyecto, persistencia SQLite, compactación de conversación con auto-summary + transcript, vista referenciada para evitar doble almacenamiento completo.
8. **UI "IDE-like" glassmorphism:** ventana sin decoración + transparente, temas claro/oscuro, paneles especializados (`ChatArea`, `TerminalInput`, `AgentPanel`, `TodoPanel`, `ExecutionPanel`, `CheckpointTimeline`, `SnapshotTimeline`, `DiffViewer`, `LivePreview`, `SettingsPanel`).
9. **Sistema de skills formal** (carpeta `skills/` con `SKILL.md` por skill, cargador `load_all_skills()` con YAML frontmatter) — convergence con Claude Code / Hermes Agent / Aithera V0.85 Memory.
10. **Tool selection por intent refinado (4 niveles)**: CHAT (no tools) / MEMORY_QUERY (solo query) / PROJECT_ACTION (núcleo + progresive disclosure via `search_tools`) / SUBAGENT (subset restringido). Más fino que el doble eje declarado al usuario.
11. **Model registry declarativo** (`src-tauri/model_registry.json`): capabilities por modelo (thinking, vision, contexto, single-turn max). Patrón借鉴 para Aithera `ai_models.py`.

#### 3. Clasificación de intención en cascada de 3 capas (`src-tauri/src/core/intent/mod.rs:55-77`)

pub async fn classify_intent(...) -> String {
    // 第一层：纯规则匹配 (Primera capa: matching puro por reglas)
    let rule_intent = classify_by_rules(msg);
    if rule_intent != Intent::Unclear {
        return rule_intent.as_str().to_string();
    }
    // 第二层：从历史消息中提取最近 4 条助手行为特征 (Segunda capa: extraer últimas 4 acciones del asistente del historial)
    let recent_assistant_actions: Vec<LastAssistantAction> = history
        .iter().rev()
        .filter_map(|m| match m {
            Message::Assistant { content } => Some(analyze_last_assistant_message(...)),
            _ => None,
        })
        .take(4)
        .collect();
    let context_intent = classify_with_context(msg, &recent_assistant_actions);
    if context_intent != Intent::Unclear { return ...; }
    // 第三层：规则和上下文均无法判定，调用轻量 LLM 兜底 (Tercera capa: LLM ligero de fallback)
    classify_intent_by_llm(client, api_key, base_url, model_id, api_format, msg, history).await
}

    const type  = (error as any).type  || "";
    const code  = (error as any).code  || "";

## Conflictos / discrepancias entre fuentes detectadas

- **Eje doble vs Tool-by-intent (4 niveles)**: el README y `pipeline.rs` declaran `audience/work_mode` como doble eje del comportamiento, pero `PROJECT_STRUCTURE.md` §10 muestra que internamente se selecciona el set de tools por `intent ∈ {CHAT, MEMORY_QUERY, PROJECT_ACTION, SUBAGENT}`, NO por `audience/work_mode`. Conclusión probable: `audience/work_mode` es la UX del usuario, `intent` es el dispatch interno. El doc final debe explicar ambos niveles.
- **"Inspirado en Claude Code" ↔ perfil independiente**: el README agradece a Claude Code explícitamente. El perfil del autor muestra 0 followers y 3 repos (ningún fork de Anthropic). Probable:借鉴 conceptual por lectura del blog de Claude Code + blog de Anthropic, no contribución directa upstream.
- **MIT declarada ↔ LICENSE ausente**: el README declara MIT pero la API devuelve `null` y no hay archivo LICENSE en la raíz. Caveat textual obligatorio (no asumible como OSS ejecutable legalmente).
- **Documentación exhaustiva ↔ silencio de actividad**: `README.md` 17 KB y `PROJECT_STRUCTURE.md` 20 KB son densidad documental comparable a proyectos OSS serios. Pero 51 días sin push + 1 fork + 4 estrellas no son consistentes con la documentación. Hipótesis: el autor dedicó ~3 semanas de energía en abril-mayo 2026 a construir y documentar, luego pausó. La sobresofisticación documental no se tradujo en adopción.
- **20+ LLMs ↔ model_registry parcial**: el README dice "20+ LLMs", pero `model_registry.json` (mencionado en `PROJECT_STRUCTURE.md` §16) parece cubrir solo 2 modelos con capabilities detalladas. Caveat: el resto de los 20+ se configura por perfil sin metadata estructurada.
